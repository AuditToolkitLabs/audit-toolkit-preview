# 5. End-User Guide

*ISO/IEC 20000-1 clauses 8.2, 8.3*

## 5.1 Accessing the application

Open your browser and navigate to `https://<your-toolkit-hostname>/`.
You will be prompted to sign in. Use the username and password provided
by your Application Administrator, or your organisation's SSO
credentials if single sign-on has been configured.

> **First login:** If this is the very first sign-in to a fresh
> installation, you will be asked to accept the End-User Licence
> Agreement. The acceptance is recorded with your username, timestamp,
> client IP address, and a SHA-256 hash of the agreement text.

## 5.2 Dashboard overview

After signing in you will see the main dashboard. Key areas are:

| Area | Description |
| --- | --- |
| **Summary cards** | Total servers, passing checks, failing checks, and pending scans at a glance. |
| **Recent audit results** | Latest completed audits listed by target host. |
| **Findings by severity** | Bar chart showing PASS / WARN / FAIL / SKIP counts. |
| **Scheduled scans** | Upcoming and in-progress scheduled runs. |
| **Alerts** | Active alerts generated by threshold violations or failed checks. |

## 5.3 Running an on-demand audit

1. Navigate to **Targets** and select the host you want to audit.
2. Click **Run Audit Now**.
3. Choose the audit profile or individual scripts to run.
4. Click **Start**. The audit runs in the background.
5. When complete, click **View Results** to see the findings.

Alternatively, use the **Orchestrator** view to run an audit across
multiple targets simultaneously.

## 5.4 Interpreting audit results

Each audit check produces one of four result codes:

| Code | Meaning |
| --- | --- |
| **PASS** | The check passed. The setting or state is compliant. |
| **WARN** | The check found a condition that may require attention but is not a hard failure. |
| **FAIL** | The check found a non-compliant or potentially insecure condition. Action is recommended. |
| **SKIP** | The check was not applicable to this target (e.g. a systemd check on an OpenRC system). |

Findings are grouped by domain (e.g. *Platform baseline*, *Network*,
*Web services*) and each finding includes:

- The check name and a brief description.
- The actual value or state observed.
- The expected value or recommendation.
- The severity and a reference to the relevant CIS Benchmark or NIST
  control where applicable.

## 5.5 Downloading and sharing reports

1. From the audit result view, click **Export**.
2. Choose the format: **PDF**, **CSV**, or **JSON**.
3. The file is generated and offered for download.

Reports can also be delivered automatically by email or webhook. Ask
your Application Administrator to configure a scheduled report.

## 5.6 Understanding the security score

The overall security score for a host is calculated as:

$$\text{Score} = \frac{\text{PASS checks}}{\text{Total applicable checks}} \times 100$$

SKIP checks are excluded from both the numerator and denominator.
WARN checks count as half a failure. Scores are shown as a percentage
and as a letter grade (A+ ≥ 95%, A ≥ 90%, B ≥ 80%, C ≥ 70%, F < 70%).

## 5.7 Finding historical results

Navigate to **History** to see all past audit runs for a target.
Results are retained according to the data-retention policy configured
by your Application Administrator (default: 90 days).

## 5.8 Managing notification preferences

If your account has permission, navigate to **Profile → Notifications**
to configure:

- Which finding severities trigger email alerts.
- A digest schedule (immediate, hourly, or daily).
