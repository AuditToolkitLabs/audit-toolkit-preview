Developer Reference

Documentation Tools & Operations

Guides for managing documentation, validating remote availability, configuring source resolution, and troubleshooting documentation access.

Overview

This reference documents the tools and operations available for managing AuditToolkit Labs documentation.

Documentation Sources

Documentation is served from a governed central source:

  • Remote (central) — From the configured central documentation mirror

The source configuration is controlled in docs-source-config.json and is enforced as remote-only for this site.

Documentation Viewer

The documentation viewer at doc-viewer.html provides an interactive documentation browser with advanced features.

📍 Source Mode Badge

Each page displays a colored badge indicating where the documentation is being served from:

  • Source: remote — Served from remote mirror
  • Source: remote (canary) — Served from remote in test mode
🐛 Debug Context Helper

Click Copy debug context to copy JSON containing:

  • Requested vs. resolved document source
  • Source mode and canary parameters
  • Effective configuration (all fields)
  • Page URL and timestamp

Use this for troubleshooting documentation access issues and support triage.

⏱️ Resolution Log

Click Show resolution log to display live source resolution steps with millisecond timings:

  • Doc load started
  • Config loaded
  • Content resolved from [local|remote]
  • Parsing, rendering, and link rewriting steps
  • Total time elapsed

Use this to identify performance bottlenecks and debug fetch failures.

Query Parameters

The viewer supports the following query parameters to control source resolution:

Parameter Value Effect
src customer-docs/... Request a specific path resolved from the central docs source
canary remote-only Force remote-only mode (when enabled)

Example: doc-viewer.html?src=customer-docs/audit-toolkit/README.md&canary=remote-only

Remote Availability Validation

Remote availability checks validate that central documentation sources resolve correctly for documentation viewer paths.

⏰ Scheduled Validation

Remote canary checks run as needed:

  • Manual trigger — Run the canary smoke workflow via GitHub Actions
  • Pre-merge validation — Pull request checks validate remote docs resolution

Source Configuration

Documentation source resolution is controlled by docs-source-config.json:

{
  "mode": "remote-only",
  "localDocsRoot": "customer-docs",
  "remoteMirrorBaseUrl": "https://raw.githubusercontent.com/AuditToolkitLabs/releases/main",
  "enableRemoteFallback": false,
  "enableCanaryRemoteOnly": false,
  "canaryQueryParam": "canary",
  "canaryQueryValue": "remote-only"
}

Configuration Fields

Field Purpose
mode Source policy (set to remote-only for this site)
localDocsRoot Logical root prefix used to resolve document paths
remoteMirrorBaseUrl Base URL for central remote documentation source
enableRemoteFallback Retained for compatibility; disabled in remote-only mode
enableCanaryRemoteOnly When true, ?canary=remote-only parameter enables remote-only testing
canaryQueryParam & canaryQueryValue Query parameter name and value to trigger canary mode

Troubleshooting

Documentation Not Loading

Check:

  1. Click Copy debug context in the doc viewer
  2. Verify resolvedSrc matches expected source
  3. Check sourceKind (remote or remote-canary)
  4. Review Show resolution log for fetch errors

Wrong Source Being Used

Check:

  1. Verify docs-source-config.json mode setting
  2. Check active query parameters in URL
  3. Inspect source mode badge color
  4. Look at resolution log to see why source was selected

Documentation Out of Sync

Check:

  1. Confirm the requested central path exists at the configured remote source
  2. Use Copy debug context and Show resolution log for triage details
  3. Review docs-source configuration and remote mirror URL

Remote Docs Unavailable

Check:

  1. Verify remote mirror URL in config is accessible
  2. Enable canary testing: Set enableCanaryRemoteOnly: true in config
  3. Test with ?canary=remote-only query parameter
  4. Review CI smoke test results in GitHub Actions
